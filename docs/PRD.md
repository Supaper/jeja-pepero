# PRD — 게시판 모니터링 & 리포팅 자동화 (jeja-pepero)

> 작성일: 2026-06-27 · 상태: Draft v0.1
> 목적: 현재 Google Sheets + Apps Script + Gmail 로 운영 중인 게시판 모니터링 워크플로를
> 독립 실행 가능한 프로그램으로 재구축하기 위한 제품 요구사항 정의.

---

## 1. 배경 (Background)

특정 웹 게시판(`thelifechurch.kr`)에 지정된 작성자들이 올리는 글(예: 큐티나눔, 기도제목 등)을
매일 모니터링하여, 새 글을 수집·기록하고 담당자에게 이메일로 보고하고 있다. 또한 매월 초에는
지난달의 큐티나눔 작성 횟수를 집계하여 **완주(달성)율** 리포트를 발송한다.

현재는 구글 스프레드시트와 Apps Script(`monitorDailyCollectionOnly`, `sendMonthlyQTReport`)로
구현되어 있으나 다음과 같은 한계가 있다.

- **확장성 부족**: 게시판/대상 그룹이 늘어날수록 스크립트와 스프레드시트가 분산되어 관리가 어렵다.
  (현재도 일일 수집용/월간 큐티용으로 스프레드시트 ID와 대상 명단이 코드에 하드코딩되어 분리 운영 중)
- **카테고리 분류 부재**: "공지" 제외와 "큐티나눔" 단순 문자열 매칭에 의존한다. 글의 카테고리를
  체계적으로 분류·집계하는 구조가 없다.
- **상태 관리 취약**: 마지막 수집 기준점을 `ScriptProperties`의 "마지막 제목" 한 건으로만 관리해
  제목 중복/수정 시 누락·중복 위험이 있다.
- **문서화 미지원**: 수집 결과를 한글(.hwpx/.hwp) 양식 문서로 만들어야 하는 요구가 있으나
  Apps Script로는 불가능하다.
- **테스트/버전관리 부재**: 코드 리뷰, 테스트, 변경 이력 추적이 어렵다.

## 2. 목표 (Goals) / 비목표 (Non-Goals)

### 2.1 목표
1. 지정한 게시판을 주기적으로 모니터링하여 **지정된 작성자**의 새 글을 안정적으로 수집한다.
2. 수집한 글을 **카테고리별로 분류**하여 저장한다.
3. **일별 알림**(새 글 요약)과 **월별 통계**(카테고리별 작성 횟수/달성률)를 이메일로 발송한다.
4. (장기) 수집·집계 결과를 **한글(.hwpx/.hwp) 양식 문서**로 자동 생성하여 보기 쉽게 문서화한다.
5. 게시판·대상 명단·카테고리 등 **설정을 코드와 분리**하여 비개발자도 운영할 수 있게 한다.

### 2.2 비목표
- 게시판에 글을 **작성/수정/삭제**하는 기능 (읽기 전용 모니터링만 수행).
- 실시간(초/분 단위) 모니터링. 일 1~수회 배치로 충분하다.
- 게시판 측 인증이 필요한 비공개 게시물 수집(초기 범위 외).
- 범용 SNS/포털 크롤러. 대상은 정해진 게시판 어댑터로 한정한다.

## 3. 사용자 (Users)

| 역할 | 설명 | 주요 니즈 |
|------|------|-----------|
| 운영 담당자 | 모니터링 결과를 보고받고 명단/설정을 관리 | 매일 새 글 요약, 매월 통계, 손쉬운 명단 수정 |
| 리포트 수신자 | 통계/문서를 전달받는 책임자 | 보기 쉬운 요약 메일, 한글 양식 문서 |
| 개발자/유지보수자 | 프로그램 운영·확장 | 테스트 가능한 코드, 명확한 설정, 로그/관측성 |

## 4. 현행 동작 분석 (As-Is, Apps Script 기준)

| 항목 | 현행 구현 |
|------|-----------|
| 대상 게시판 | `thelifechurch.kr` `boardID=www56`, 작성자명(`keyfield=name`)으로 검색 |
| 대상 작성자 | 코드 내 `targetNames` 배열 (일일/월간 각각 별도 명단) |
| 수집 범위 | `START_DATE_STRING`(예: `2026.02.01`) 이후 글, "공지" 포함 글 제외 |
| 중복 방지 | 작성자별 `LAST_TITLE_<name>` 프로퍼티에 최신 글 제목 1건 저장, 동일 제목 만나면 중단 |
| 저장 | 작성자명 시트에 `[수집일시, 작성일, 제목, 링크]` append |
| 일별 알림 | 새 글이 있을 때 작성자별 목록을 HTML 메일로 발송 |
| 월별 통계 | 지난달, 제목에 `큐티나눔` 포함 글 수 집계 → `count/일수` 달성률(%) 색상 표기 메일 |
| 서버 부하 방지 | 작성자 요청 사이 `Utilities.sleep(1000)` 1초 대기 |

> To-Be에서는 위 동작을 보존하되, 카테고리 분류·견고한 상태관리·문서화·설정 분리를 추가한다.

## 5. 기능 요구사항 (Functional Requirements)

### FR-1. 게시판 모니터링 & 수집
- FR-1.1 설정된 게시판에서 **지정 작성자별**로 글 목록을 조회한다.
- FR-1.2 게시판별 파싱 로직은 **어댑터(adapter)** 로 분리하여 신규 게시판 추가가 용이해야 한다.
- FR-1.3 요청 간 **정중한 지연(rate limit)** 과 재시도(backoff)를 적용해 대상 서버 부하를 최소화한다.
- FR-1.4 각 글에서 최소 **작성일·제목·링크·작성자**를 추출한다. (가능 시 게시글 ID도 추출)

### FR-2. 카테고리 분류
- FR-2.1 제목/내용에 대한 **규칙 기반 분류기**(키워드/정규식)로 글을 카테고리에 매핑한다.
  - 예: 제목에 `큐티나눔` → `QT`, `기도` → `기도제목`, `공지` → 제외/`NOTICE`.
- FR-2.2 분류 규칙은 설정 파일로 관리하며, 미분류 글은 `기타(UNCATEGORIZED)`로 둔다.
- FR-2.3 "공지" 등 **제외 규칙**을 카테고리 규칙으로 일반화한다.

### FR-3. 중복 방지 & 상태 관리
- FR-3.1 수집한 글은 **게시글 식별자(링크/ID 우선, 없으면 작성자+작성일+제목 해시)** 로 dedupe 한다.
- FR-3.2 "마지막 제목 1건" 방식 대신, 영속 저장소에서 **이미 수집된 글 집합**을 기준으로 신규 여부를 판단한다.
- FR-3.3 수집 기준일(start date) 이전 글은 건너뛴다.
- FR-3.4 상태 초기화(reset) 기능을 제공한다. (현행 `resetMemory` 대체)

### FR-4. 저장 (Persistence)
- FR-4.1 수집 글을 구조화 저장소(초기: SQLite, 선택: Google Sheets 동기화)에 보관한다.
- FR-4.2 스키마: `post(id, source, author, posted_date, title, url, category, collected_at)`.
- FR-4.3 (호환) 기존 Google Sheets 출력 포맷을 옵션으로 유지/동기화할 수 있다.

### FR-5. 일별 알림
- FR-5.1 신규 수집 글을 **작성자별·카테고리별**로 묶어 요약 메일을 발송한다.
- FR-5.2 새 글이 없으면 메일을 보내지 않거나, "변동 없음" 다이제스트를 옵션으로 제공한다.
- FR-5.3 메일 본문은 HTML 템플릿으로 관리한다.

### FR-6. 월별 통계
- FR-6.1 지난달(또는 지정 기간)의 **작성자 × 카테고리** 작성 횟수를 집계한다.
- FR-6.2 카테고리별 **달성률(작성수/기준일수 등)** 을 계산하고 임계값 구간별 색상으로 강조한다.
  (현행: ≥90% 파랑 / ≥70% 초록 / ≥50% 주황 / 그 외 빨강)
- FR-6.3 통계 리포트를 메일로 발송하고, 데이터(CSV/표)를 산출물로 남긴다.

### FR-7. 한글 문서 자동화 (장기 / Phase 3)
- FR-7.1 미리 정의한 **.hwpx 템플릿**에 집계 결과를 채워 문서를 생성한다.
- FR-7.2 .hwpx(개방형 XML/zip) 기반 치환을 1차 목표로, .hwp(바이너리)는 변환/COM 자동화로 확장한다.
- FR-7.3 생성 문서를 메일 첨부 또는 지정 경로/드라이브에 저장한다.

### FR-8. 설정 & 운영
- FR-8.1 게시판 URL, 대상 명단, 카테고리 규칙, 발송 대상, 스케줄을 **설정 파일/환경변수**로 분리한다.
- FR-8.2 자격증명(메일 등)은 코드에 하드코딩하지 않고 환경변수/시크릿으로 주입한다.
- FR-8.3 스케줄링은 cron/GitHub Actions 등으로 외부화 가능해야 한다.

## 6. 비기능 요구사항 (Non-Functional)

- **신뢰성**: 단일 작성자 수집 실패가 전체 배치를 중단시키지 않는다(현행과 동일하게 격리).
- **관측성**: 구조화 로그와 실행 요약(수집 건수, 실패 작성자)을 남긴다.
- **멱등성**: 같은 배치를 재실행해도 중복 저장/중복 메일이 발생하지 않는다.
- **이식성**: 로컬/서버/CI(GitHub Actions) 어디서든 동일하게 동작한다.
- **정중한 크롤링**: rate limit, 재시도, User-Agent 명시, robots 존중.
- **보안/개인정보**: 수집 대상은 작성자명·제목·링크 등 공개 게시 정보로 한정. 자격증명은 시크릿 관리.

## 7. 데이터 모델 (초안)

```
Source        # 모니터링 대상 게시판
  id, name, base_url, board_id, adapter, enabled

Author        # 지정 작성자
  id, source_id, name, group, enabled

Post          # 수집 글
  id, source_id, author, posted_date, title, url, post_key(unique),
  category, collected_at

Category      # 분류 규칙
  id, name, match_rules(keywords/regex), is_excluded

RunLog        # 배치 실행 기록
  id, started_at, finished_at, new_count, errors
```

`post_key`는 URL/게시글 ID 우선, 없으면 `hash(author + posted_date + title)`.

## 8. 아키텍처 (제안)

권장 스택: **Python** (HTML 파싱·HWPX(zip/XML) 조작·스케줄링 생태계가 풍부).

```
fetcher(adapters/) → parser → categorizer → store(SQLite) →
  ├─ notifier(daily email)
  ├─ reporter(monthly stats + email)
  └─ doc-generator(hwpx, Phase 3)
config(yaml/.env) 가 전 구성요소를 주입, scheduler(cron/Actions)가 트리거
```

- `adapters/`: 게시판별 fetch+parse (1차: thelifechurch `www56`).
- `categorizer`: 규칙 기반 분류(설정 파일).
- `store`: SQLite 기본, Google Sheets 동기화는 선택 모듈.
- `notifier`/`reporter`: HTML 템플릿 기반 메일.
- `doc-generator`: hwpx 템플릿 치환.

## 11. 위험 및 가정 (Risks & Assumptions)

- **R1 게시판 HTML 변경**: 마크업 변경 시 파서가 깨질 수 있음 → 어댑터 격리 + 파싱 실패 알림.
- **R2 .hwp 바이너리**: 순수 자동화가 어려움 → .hwpx(XML) 우선, .hwp는 변환/COM 경로로 분리.
- **R3 메일 발송 한도/스팸**: 발송량/도메인 정책 확인 필요.
- **A1**: 대상 게시물은 인증 없이 접근 가능한 공개 정보.
- **A2**: 작성일이 게시판에 `YYYY.MM.DD` 형식으로 노출됨(현행 파싱 가정 유지).

## 12. 미해결 질문 (Open Questions)

- 운영 환경: 로컬 cron / 상시 서버 / GitHub Actions 중 무엇을 기본으로?
- 저장소: SQLite 단독으로 충분한가, 기존 Google Sheets와의 양방향 동기화가 필요한가?
- 메일 발송: 기존 Gmail 계정 유지 vs SMTP/메일 API 전환?
- 한글 문서: 최종 산출물이 .hwpx로 충분한가, .hwp 호환이 필수인가? 표준 양식(템플릿) 확보 여부?
